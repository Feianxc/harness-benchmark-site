# OHBP 低成本真实性与上传规则

## 1. 总目标

在你预算有限的情况下，真实性不能主要依赖：

- 大规模全量人工复核
- 高算力全量复跑
- 重型专有基础设施

而应该主要依赖：

1. **上传前预注册**
2. **上传时证据链**
3. **服务端复算**
4. **少量受控抽样复跑**
5. **严格分层发布**

---

## 2. 推荐的低成本真相链

## 2.1 第一步：预注册

只要不是纯 Community，建议要求：

- `study_id`
- `run_group_id`
- 计划上传的 attempt 数
- benchmark / lane / execution contract
- requested trust tier

目的：

- 防止先跑十几次再只挑最好的一次上传

---

## 2.2 第二步：上传对象

建议必须上传：

1. `manifest.json`
2. `checksums`
3. `public bundle`
4. `sealed audit bundle`（高信任层必需）
5. `task-level results`
6. `trace / interaction log`
7. `execution artifact ref`
   - 代码 commit
   - container digest
   - 依赖锁文件

不建议接受：

- 只传一个摘要 JSON
- 只传一个 score
- 只传截图

---

## 2.3 第三步：服务端复算

上传后平台必须自己重算：

- public bundle digest
- sealed bundle digest
- aggregate metrics
- run-group 完整性
- manifest 与 trace / task-results 的一致性

这里的核心原则是：

> **平台相信 bundle 真相，不相信客户端自报。**

这也是你们当前 `mock-intake` 已经在做的正确方向。

---

## 2.4 第四步：平台抽样复跑

因为你预算有限，所以不建议：

- 每条都全量复跑

建议做三种低成本复跑：

### A. 头部复跑

只对以下条目做：

- 冲榜条目
- 进入 Top N 的条目
- 大幅超越历史表现的条目

### B. 随机抽样复跑

每周或每批次随机抽几个。

目的：

- 让提交者知道平台会抽检

### C. 争议复跑

被举报、异常、可疑时触发。

---

## 3. 真实性的“六道闸门”

真实性判定不建议做成一个黑盒总分，而建议做成六道闸门。

## 闸门 1：文件真

检查：

- manifest 在不在
- checksums 对不对
- public_bundle_digest 是否可复算
- sealed_audit_bundle_digest 是否匹配

未通过：

- 直接不能进入高信任层

## 闸门 2：次数真

检查：

- declared attempt total
- observed attempt total
- 是否缺 attempt
- 是否只上传最佳一次

未通过：

- 不能进入 Reproduced / Official

## 闸门 3：人工真

检查：

- declared autonomy mode
- interaction log
- approval events
- manual edit / freeform input 痕迹

未通过：

- 降级到 assisted / interactive 层

## 闸门 4：环境真

检查：

- execution_contract_digest
- tool policy
- network policy
- runtime / container / lockfile

未通过：

- 不得混入可比排行榜

## 闸门 5：泛化真

检查：

- `benchmark_tuned_flag`
- public / hidden split
- 是否有 sealed evidence
- 是否 benchmark-specific

未通过：

- 不能当作 general-purpose 官方结果

## 闸门 6：稳定真

检查：

- `n_runs`
- repeated-run protocol
- uncertainty 是否可展示

未通过：

- 不得进入 Official Verified

---

## 4. 低成本条件下的 trust tier 规则

## 4.1 Community Lab

建议门槛：

- 通过闸门 1
- 基本通过闸门 2
- 有最小证据链

但：

- 不要求平台复跑
- 不进入默认官方主榜

## 4.2 Reproducibility Frontier

建议门槛：

- 通过闸门 1 到 5
- 至少 3 runs
- evidence 比较完整

但：

- 平台可以尚未完成 full official audit

## 4.3 Official Verified

建议门槛：

- 六道闸门全部通过
- 至少 5 runs
- 有一次平台抽样复跑 / hidden subset 复核
- 无 unresolved dispute

---

## 5. 为什么这种方案适合单人运营

因为它把资源花在了最值钱的地方：

### 花小钱的部分

- schema / digest / bundle 校验
- completeness proof
- metadata consistency
- evidence gating

### 花中等钱的部分

- 少量 official lane
- 少量 hidden subset
- 少量随机抽检

### 不做的大钱部分

- 所有 benchmark 全量复跑
- 所有条目长期重跑
- 大规模 live environment 审计

---

## 6. 推荐的资源优先级

## P0：必须先做

1. platform-side artifact store
2. bundle truth 覆盖客户端声明
3. attempt 完整性检查
4. evidence layer 禁止 demo fallback 冒充真条目

## P1：下一步做

1. 头部条目抽样复跑
2. hidden subset 检查
3. dispute / corrected / invalidated 状态机

## P2：预算允许再做

1. 更强 attestation
2. 更复杂 environment report
3. 更多 benchmark lane

---

## 7. 推荐的简单运营规则

为了省钱，我建议再加 5 条运营规则：

1. **每个 system fingerprint 每周最多允许一次高信任提交**
2. **同一 benchmark version 下限制重复刷榜频率**
3. **Official 只接受少量 lane**
4. **只对 Top entries 和异常 entries 做复跑**
5. **Community / Frontier / Official 三层绝不混榜**

---

## 8. 一句最重要的话

> **低成本不等于低可信；真正的关键不是“多花钱重跑所有东西”，而是“先把不可信的东西挡在 Official 门外”。**
