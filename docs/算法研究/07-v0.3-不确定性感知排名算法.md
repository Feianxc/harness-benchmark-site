# OHBP v0.3 不确定性感知排名算法

## 1. 结论

v0.3 的排名算法不再直接按裸成功率硬排，而采用：

> **固定 slice 内比较 + 任务成功率 + Wilson 95% 下界保守排序 + rank band 展示。**

一句话解释：

> **成功率高不代表一定更强；样本少时要把“不确定性”算进去。**

---

## 2. 为什么要这样改

如果 A 做了 10 道题，成功 8 道，成功率是 80%。  
B 做了 100 道题，成功 75 道，成功率是 75%。

如果只看裸成功率，会觉得 A 更强。

但从第一性原理看：

- A 只有 10 个样本，可能只是运气好；
- B 有 100 个样本，更稳定；
- 排行榜真正要回答的不是“谁当前点估计更高”，而是“我们有多确定它真的更强”。

所以 v0.3 使用 Wilson 区间：

- 区间下界越高，代表“保守看也不错”；
- 样本越少，下界自然越低；
- 这样可以低成本惩罚小样本虚高。

---

## 3. 当前实现范围

v0.3 当前只改 **Evidence Board**。

不改：

- consumer curated guide
- host-fit demo
- compare 静态策展分

原因：

> consumer 层仍然是导购层，不是真实 verifier-backed 真榜。

---

## 4. 输入字段

当前最小实现使用已有字段：

- `success_rate`
- `n_tasks`
- `n_runs`
- `task_package_digest`
- `execution_contract_digest`
- `tolerance_policy_digest`
- `median_cost_usd`
- `p95_latency_ms`

当前没有持久化 task-level `success_count`，所以先做保守近似：

```txt
observed_successes = round(success_rate * n_tasks)
effective_n = n_tasks
```

注意：

> 当前没有直接用 `n_tasks * n_runs` 当样本数。

原因是：

- 多次 run 可能跑的是同一批任务；
- 同一任务的重复运行不是完全独立样本；
- 直接乘起来会制造假的精度。

---

## 5. Wilson 95% 区间

设：

```txt
x = 成功数
n = 样本数
p = x / n
z = 1.96
```

公式：

```txt
denominator = 1 + z² / n
center = (p + z² / (2n)) / denominator
half = z * sqrt((p(1-p) + z²/(4n)) / n) / denominator

lower = center - half
upper = center + half
```

v0.3 排序分：

```txt
ranking_score = lower
```

也就是：

> 用 Wilson 95% 下界排序。

---

## 6. 排序规则

在同一个 fixed slice 内：

0. 先保证是同一个 `task_package_digest`、同一个执行合约、同一个容差策略；不同任务包不能混排
1. 先按 `WilsonLower95` 降序
2. 再按裸 `success_rate` 降序
3. 再按 `median_cost_usd` 升序
4. 再按 `p95_latency_ms` 升序

这符合 v0.2 的原则：

- success-first
- cost / latency 只做 tie-break
- 不做神秘加权总分

---

## 7. rank band 怎么来

v0.3 默认展示：

- `置信档 A`
- `置信档 B`
- `置信档 C`

而不是默认展示：

- `#1`
- `#2`
- `#3`

规则：

如果两个相邻条目的 Wilson 95% 区间重叠，就放在同一个 band。

只有当领先者的下界高于下一名的上界时，才说明差距比较清楚。

更具体地说，v0.3 使用“相邻重叠链”：

- A 和 B 重叠；
- B 和 C 重叠；
- 即使 A 和 C 不直接重叠；
- A/B/C 仍先放在同一个置信档。

这样做更保守，避免因为第一个条目和第三个条目不重叠，就忽略中间条目的连接作用。

---

## 8. 什么时候允许硬名次

当前实现里，硬名次必须满足：

1. active eligible entries 至少 3 条；
2. 每条 `effective_n >= 30`；
3. 相邻条目的 95% Wilson 区间完全分离。

否则 Board 继续保持：

```txt
ranked_tiered
```

而不是：

```txt
ranked_ordinal
```

这就是 fail-closed：

> 不确定就分档，不硬排。

---

## 9. 前端现在展示什么

Board entry 现在会带：

- 成功率
- `95% CI`
- `rank_band`
- `rank_confidence`（当前按单条结果的区间宽度与样本数估计，更准确地说是“区间精度”，不是硬名次保证）
- `effective_n`
- `observed_successes`

Board 页面会显示排序策略：

- 方法：`wilson_lower_bound_success_rate_v0_3`
- 置信水平：95%
- 硬名次门槛：`n >= 30`
- 是否允许硬名次
- 分离规则说明

---

## 10. 为什么适合单人低成本运营

这套算法只需要：

- 成功数
- 任务数
- 成功率

不需要：

- 昂贵复跑
- 大规模人工审核
- 复杂贝叶斯建模
- Elo / TrueSkill 对战数据

它能立刻解决低预算阶段最危险的问题：

> 小样本高成功率把榜单骗上去。

---

## 11. 当前不要做什么

### 不做 Elo / TrueSkill

因为 OHBP 现在不是 pairwise battle 数据。

### 不做加权总分

不要写：

```txt
score = 0.7 * success + 0.2 * cost + 0.1 * latency
```

原因：

- 权重主观；
- 会制造争议；
- 会掩盖成功率的不确定性。

### 不直接把重复运行乘成样本数

不要默认：

```txt
effective_n = n_tasks * n_runs
```

除非后续真的有 task-level / attempt-level 独立结果。
